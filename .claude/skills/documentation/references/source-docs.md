# Source Documentation Conventions

Conventions for Sphinx documentation in `docs/source/`.

## File Organization

```
docs/source/
├── index.md                    # Landing page
├── getting_started/            # Installation, quickstart
├── user_guide/                 # Conceptual guides by method
├── development/                # Contributor docs
├── reference/                  # API reference (RST)
│   ├── index.rst
│   ├── models/                 # Method subpackages (gfa/, mdlag/)
│   ├── observation/            # Observation model components
│   ├── state/                  # State model components
│   └── ...
└── auto_examples/              # Generated by sphinx-gallery
```

## Title Conventions

Use **sentence case** for all titles (capitalize first word and proper nouns only).

| Page Type | Convention | Example |
|-----------|------------|---------|
| Top-level index | Sentence case, descriptive | "API reference" |
| Category index | Sentence case, descriptive | "Observation model" |
| Method index | Full name with abbreviation | "Group factor analysis (GFA)" |
| Module page | Module path, lowercase | "gfa.config" |

**Rationale**: Sentence case is the modern convention used by NumPy, pandas, scikit-learn, and most scientific Python projects.

## Cross-Reference Style

Use Sphinx cross-references for internal links. Reserve raw URLs for truly external resources.

| Link Type | Format | Example |
|-----------|--------|---------|
| Internal doc | MyST `{doc}` or RST `:doc:` | `{doc}`../user_guide/gfa`` |
| Internal ref | MyST `{ref}` or RST `:ref:` | `{ref}`gfa-user-guide`` |
| API class | `:class:` | `:class:`GFAModel`` |
| API function | `:func:` with full path | `:func:`~latents.gfa.inference.fit`` |
| External URL | Markdown link | `[paper](https://doi.org/...)` |

**Rationale**: Cross-references survive file renames and catch broken links at build time.

## RST Index Pages

Index pages introduce a section and list child pages via `toctree`.

### Category Index

For categories like `models/`, add a brief introductory paragraph:

```rst
.. _models:

Models
======

Model implementations for specific latent variable methods.

.. toctree::
    :maxdepth: 2

    gfa/index
    mdlag/index
```

### Subpackage Index

For subpackages like `observation/`, add explicit introductory text:

```rst
.. _observation:

observation
===========

Observation model components organized by probabilistic level.

.. toctree::
   :maxdepth: 1

   priors
   posteriors
   realizations
```

**Note**: Do NOT use `.. automodule::` on index pages. While it avoids duplicating the `__init__.py` docstring, it causes unintended side effects: creating API entries based on exports and adding to the documentation structure. Explicit prose gives full control over index page content.

## RST Module Pages

Module pages document all public classes and functions.

### Standard Structure

```rst
.. _module_label:

module.name
===========

.. automodule:: latents.module.name
   :no-members:

.. currentmodule:: latents.module.name

.. rubric:: Classes

.. autosummary::
   :nosignatures:

   ClassName1
   ClassName2

.. rubric:: Functions

.. autosummary::
   :nosignatures:

   function_name

----

.. autoclass:: ClassName1
   :members:

.. autoclass:: ClassName2
   :members:

.. autofunction:: function_name
```

### When to Use Summary Tables

For modules with multiple items, include `rubric` + `autosummary` tables at the top for quick navigation. Single-item modules can skip the summary table.

### Grouping in Summary Tables

**Default**: Group by object type (`Classes`, `Functions`).

**Semantic grouping**: When a module has logical sub-categories:

```rst
.. rubric:: Callbacks

.. autosummary::
   :nosignatures:

   ProgressCallback
   LoggingCallback

.. rubric:: Utilities

.. autosummary::
   :nosignatures:

   invoke_callbacks
```

## Class Documentation

### Standard Classes

Classes with methods to document:

```rst
.. autoclass:: ClassName
   :members:
```

### Dataclasses

Dataclass fields are documented in the Parameters section of the class docstring. Using `:members:` causes redundant field documentation in both Parameters and Attributes sections.

**Pure data dataclasses** (no public methods beyond properties):

```rst
.. autoclass:: GFAFitConfig
   :no-members:
```

**Dataclasses with public methods** (e.g., `sample()`):

```rst
.. autoclass:: ObsParamsPrior
   :members:
   :exclude-members: hyperprior
```

Use `:members:` to document methods, then `:exclude-members:` to suppress the redundant field attributes.

**Rationale**: `:exclude-members:` prevents the duplicate attribute rendering that occurs when Sphinx documents dataclass fields both in Parameters (from docstring) and Attributes (from `:members:`).

### Primary User-Facing Classes

Large classes like `GFAModel` with many methods:

```rst
.. autoclass:: GFAModel
   :autosummary:
   :autosummary-nosignatures:
   :autosummary-sections: Methods
```

**Rationale**: `:autosummary:` provides a methods table for quick navigation.

## What NOT to Include

- **`:show-inheritance:`**: Most classes inherit from `object` trivially. Skip unless inheritance is semantically meaningful.
- **`:undoc-members:`**: If it's not documented, don't expose it in the API reference.

## Stub Pages

Pages for planned but unimplemented content should remain as stubs with a `toctree` only (no "coming soon" text). They will be populated in later documentation phases.
